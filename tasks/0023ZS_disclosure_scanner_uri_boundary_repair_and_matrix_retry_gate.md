## 6. Mandatory disclosure-classifier repair

The reported false positive came from treating the `s:/` substring inside `https://godotengine.org` as a drive path.

Implement one explicit helper for Windows absolute-drive-path detection. Its semantics must be equivalent to requiring a non-URI left boundary, for example:

```text
(?<![A-Za-z0-9+.\-])[A-Za-z]:[\\/]
```

The exact implementation may differ, but it must satisfy all requirements below.

### 6.1 Required detections

The drive-path classifier must detect at minimum:

```text
C:\Users\player\profile.json
D:/Sandbox/file.txt
path=E:\x\y
"F:/quoted/path"
file:///C:/Users/player/file.txt
```

`file:///C:/...` remains a disclosure because it contains a real absolute drive path.

Separate UNC-path detection must remain active and detect at minimum:

```text
\\server\share\file.txt
```

### 6.2 Required non-detections for the drive-path category

The drive-path classifier must not classify these as Windows disk paths:

```text
https://godotengine.org
http://example.invalid/a
Godot Engine v4.x - https://godotengine.org
res://scenes/app/app_root.tscn
user://delayed_self_profile.json
```

This requirement concerns the drive-path category only. Other independently frozen forbidden-disclosure categories may still reject a token such as `user://` when appropriate.

### 6.3 Anti-weakening rules

Do not:

- strip all URI tokens before scanning;
- blanket-allow every string containing `://`;
- whitelist an entire Godot banner line;
- whitelist arbitrary `http` or `https` output;
- remove UNC detection;
- downgrade disclosure failures to warnings;
- scan only selected case outputs;
- suppress stderr or raw case output before evidence capture;
- alter product output to satisfy the scanner.

The scanner must remain fail-closed for genuine drive, UNC, ordinary production path, raw Profile content, unsupported numeric-version, raw filesystem-error and other manifest-declared disclosure categories.
