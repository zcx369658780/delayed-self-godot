## 1. 游戏概述

### 1.1 暂定名称

- 中文名：**迟到的我**
- 英文工作名：**Delayed Self**
- 仓库代号：`delayed-self-godot`
- 名称状态：工作名，可在最终提交前更改；代码与资源命名优先使用 `delayed_self`。

### 1.2 一句话介绍

> 在一座被时间延迟撕裂的研究站中，你必须与数回合前的自己合作，踩下机关、开启通路，并拼回一条来得太晚的求救信息。

### 1.3 类型与平台

- 类型：2D、俯视角、网格制、回合制、延迟分身解谜
- 游戏模式：单人、离线
- 主要目标平台：Windows
- 最低投稿要求备用平台：Web（是否正式支持由导出预检决定）
- 单次完整流程：约 25–45 分钟
- 目标关卡数：
  - 8 月 10 日候选版：2 个教程关 + 6 个正式关
  - 最终提交版：2 个教程关 + 8–10 个正式关
- 推荐输入：键盘；手柄支持属于后续增强项，不得阻塞首个完整版本。

### 1.4 核心卖点

1. **“滞后”就是规则，而不是题材标签**：玩家的输入会在若干回合后由时间回声重新执行。
2. **与过去的自己合作**：玩家需要提前规划，让未来的自己利用过去留下的行动。
3. **谜题可形式化验证**：每关都能用求解器计算可解性、最短步数和解法数量。
4. **低美术成本、高表达效率**：用统一的几何/像素风格、残影、时间轨迹和音效表达时间错位。
5. **短流程但完整**：包含主菜单、设置、关卡选择、结算、存档、重开和通关结尾。

## 3. 设计支柱

### P1：先因后果，但结果迟到

每个重要结果都应让玩家理解其原因来自若干回合之前。延迟不能只是让操作变慢，而应产生计划、预判和协作。

### P2：复杂来自组合，不来自操作难度

玩家只需要移动、等待、互动和重开。难度来自时间依赖关系、机关组合和路线规划，不来自反应速度或像素级操作。

### P3：失败必须可理解、可快速重试

玩家应在数秒内明白失败原因。每关允许即时重开，推荐支持逐回合撤销；不设置长死亡动画或重复过场。

### P4：谜题必须可验证

每个正式关卡都必须通过自动求解或等价验证，至少记录：

- 是否可解；
- 最短解长度；
- 最短解数量；
- 可接受步数阈值；
- 使用了哪些机制；
- 是否存在明显跳过预期机制的捷径。

### P5：先完成一款游戏，再扩展一套系统

任何新机制必须回答：它是否能至少支撑两个高质量关卡？如果不能，不加入首发范围。

## 4. 核心玩法

### 4.1 基本回合

每个回合由玩家选择一个指令：

- 上移；
- 下移；
- 左移；
- 右移；
- 等待；
- 互动属于后续 schema 候选，schema v1 不启用；
- 重开不计入世界回合；逐步撤销属于后续产品决定。

逻辑上，玩家与所有时间回声从同一个回合起始状态解析动作并同时提交位置；随后环境结算机关、门和胜利条件。视觉动画可以依次播放，但不得改变已经解析的逻辑结果。

### 4.2 时间回声

关卡定义一个或多个延迟值，例如 `D = 3`。

在世界回合 `t`：

- 当前玩家执行本回合输入 `u_t`；
- 延迟为 `D` 的回声执行历史输入 `u_(t-D)`；
- 当 `t-D <= 0` 时，回声执行等待；
- 回声从关卡规定的出生格开始；
- 回声是玩家过去行动的可视化实体，而不是独立 AI。

### 4.3 首版碰撞与结算规则

为降低歧义，首版冻结以下规则：

Task 0002 的 schema v1 只实现玩家、时间回声、墙、压力板、组合门与玩家出口；更广的候选机制不属于首个垂直切片合同。

1. 玩家和时间回声可以占据同一格，不互相阻挡。
2. 不同时间回声可以互相重叠。
3. 玩家和回声都不能穿过墙。
4. 门在当前回合结算结束后更新，新的开闭状态从下一回合移动阶段生效。
5. 压力板由玩家或任意回声占据时激活。
6. 目标同步点要求指定数量/类型的实体在同一回合分别占据对应目标格。
7. 回声若因世界状态变化导致历史移动无效，则停留原地；历史输入不会改写。
8. 首个垂直切片不包含箱子推动，以避免同步移动、占位和死锁规则过早膨胀。
9. 危险格属于后续候选机制；schema v1 不包含危险或失败结算。
10. 每关都有即时重开；逐步撤销不属于 schema v1，并保留为后续产品决定。

这些规则必须由模拟层和求解器共享，不能在显示层复制另一套逻辑。

## 5. 形式化模型

### 5.1 关卡状态

一个状态 `S_t` 至少包含：

- 当前玩家位置；
- 各时间回声位置；
- 当前回合索引（运行时/指标字段；schema v1 solver canonical key 不包含绝对回合数）；
- 历史输入队列；
- 压力板占用状态（schema v1 由角色位置派生，不持久化到 canonical key）；
- 门等环境状态（激光、桥属于后续 schema）；
- 可互动机关状态（schema v1 不包含）；
- 收集物/叙事碎片状态（schema v1 不包含）；
- 成功标记；失败/危险状态属于后续 schema。

### 5.2 动作集合

基础动作集合：

```text
A = {UP, RIGHT, DOWN, LEFT, WAIT}
```

扩展动作：

```text
INTERACT
```

`INTERACT` 不属于 schema v1；只有在至少两个正式关卡确实需要、并经后续合同任务授权时才可能进入新版本动作集合。

### 5.3 状态转移

统一状态转移函数：

```text
S_(t+1) = F(S_t, u_t, level_rules)
```

`F` 必须是确定性的。相同初始状态和相同输入序列必须得到完全相同的结果。

### 5.4 胜利条件

Schema v1 的唯一胜利条件是当前玩家到达出口，时间回声不能触发完成。下列其他组合目标是后续 schema 候选：

- 当前玩家到达出口；
- 指定压力板在同一回合全部激活；
- 当前玩家与回声分别位于指定同步点；
- 在出口开启状态下到达出口；
- 收集必要的信号碎片后到达出口。

首发关卡避免使用模糊的隐藏胜利条件。

## 6. 难度与可解性指标

每个候选关卡都应生成一条机器可读的验证记录。

### 6.1 核心指标

- `L*`：最短通关步数；
- `N*`：最短解数量；
- `N(k)`：不超过 `L* + k` 步的解法数量；
- `B`：平均有效分支数；
- `R_dead`：搜索中死路状态占比；
- `W_req`：最短解中必须等待的次数；
- `C_echo`：必须参与胜利链的回声数量；
- `C_delay`：延迟值的种类与跨度；
- `C_chain`：机关依赖链最长长度；
- `C_mech`：关卡使用的机制数量；
- `Shortcut`：是否存在绕过预期教学目标的短解。

### 6.2 建议难度分层

| 难度 | 典型特征 |
|---|---|
| 教程 | 单一机制；最短解短；反馈直接；允许多解 |
| 简单 | 1 个回声、1 条机关依赖；需要一次主动等待 |
| 中等 | 1–2 个回声；路线与同步组合；存在可理解的失败分支 |
| 困难 | 多延迟或长依赖链；需要规划完整输入序列；仍应避免纯试错 |
| 挑战关 | 可选；允许较少解法和更高规划深度，不阻塞主线通关 |

### 6.3 关卡验收底线

正式关卡不得进入候选构建，除非：

1. 求解器找到至少一个解；
2. 人类试玩能理解目标；
3. 最短解没有明显利用 Bug；
4. 重开后状态完全复位；
5. 关卡使用的机制已在此前教学或本关安全区域解释；
6. 失败原因可从画面或提示理解；
7. 关卡数据和验证摘要已纳入版本控制。

## 8. 初始关卡结构

### 教程 0：动作会留下痕迹

- 教授移动、等待、重开；
- 玩家看到输入轨迹，但尚无回声；
- 目标：到达出口。

### 教程 1：三回合后的我

- 延迟值 `D=3`；
- 一个压力板和一扇门；
- 玩家第一次理解“先走开，回声稍后替自己踩板”。

### 关卡 2：错开的两扇门

- 一个回声；
- 两段门状态；
- 教授门从下一回合生效。

### 关卡 3：必须等待

- 最短解必须包含等待；
- 打破“每回合都移动才更快”的直觉。

### 关卡 4：两条时间线

- 两个不同延迟的回声；
- 清晰区分两种残影和时间轨迹。

### 关卡 5：迟到的开关

- 引入一次性或可切换机关；
- 强调过去输入在变化后的世界中执行。

### 关卡 6：时间锚点

- 玩家主动清空/重置历史；
- 防止无限回声堆积并创造阶段性规划。

### 关卡 7：求救信号

- 综合此前机制；
- 同步完成多个目标；
- 解锁最后一段叙事。

### 终局关：回声终于抵达

- 以较低操作压力综合主题；
- 玩家完成最终同步后，迟到多年的求救信息被完整播放；
- 结尾控制在 1–2 分钟，避免大规模过场制作。

Task 0018 将未来 sequence 9 的规划 ID 冻结为 `signal_convergence / Signal Convergence / 信号汇流`，主结构为先建立一个单机关门窗口，再以双机关 A+B 门完成最终汇流。其规划语法限定为一个 YOU、共享历史的 E2/E4、三个机关、两扇初始关闭的组合门、恰好两个可见依赖阶段和最终门后的玩家专属 EXIT。该段仅是规划引用，不代表正式关卡、目录迁移、求解结果、结尾演出或本地化已经实现。

实际关卡数量可根据垂直切片质量收缩。宁可 8 个高质量关卡，不做 15 个重复关卡。

## 9. 叙事与主题表达

### 9.1 背景

一座研究“远距离信号同步”的设施发生事故。多年后，自动修复体收到一条严重延迟的求救信息。玩家进入设施，以为自己在追踪幸存者，实际看到的“人影”是同一个维护体在不同时间片留下的行动记录。

### 9.2 叙事原则

- 文本少而精；
- 不用长对话打断解谜；
- 每两到三关获得一段信号碎片；
- 叙事信息与新机制同时出现；
- 最终情绪不是“拯救成功”，而是“迟到的回应仍然被听见”。

### 9.3 主题陈述草案

> 《迟到的我》把“滞后”变成了玩家每一步都必须面对的规则。玩家无法直接命令过去的自己，只能通过当前行动，为数回合后才抵达的时间回声留下可用的路径。机关的开启、合作的发生和求救信息的完整呈现都晚于它们的原因。游戏希望表达：有些回应来得太迟，无法改变已经发生的事情，但迟到并不等于毫无意义。

该文本可在最终提交前根据实际成品修改。

Task 0018 的有界终局表达沿用“迟到不等于毫无意义”：在普通 YOU 到达 EXIT 的完成条件之后，仅规划通过现有结算文字能力表达“信号虽迟但被听见”。具体中英文文本、演出、美术、音频和完整通关页面仍须后续单独授权。

## 10. 视觉方向

### 10.1 风格

- 2D 极简科技遗迹；
- 可采用低分辨率像素或清晰矢量风格，最终只能冻结一种；
- 角色以几何轮廓、核心光点和残影区分；
- 当前玩家清晰、回声半透明并带时间编号/轨迹；
- 机关状态必须优先保证可读性。

### 10.2 画面目标

- 16:9；
- 推荐逻辑分辨率：640×360 或 960×540，待垂直切片测试后冻结；
- Windows 窗口与全屏切换；
- UI 在常见 1080p 显示器上清晰；
- 不依赖小字号解释关键信息。

### 10.3 特效候选

- 时间回声拖影；
- 输入轨迹逐格点亮；
- 同步成功时的波纹；
- 延迟门状态变化的预告脉冲；
- 可选的轻量全屏时间噪声 Shader。

Shader 是加分项，不得阻塞玩法和稳定性。

## 11. 音频方向

- 当前玩家动作：直接、清晰、较近；
- 回声动作：延迟、滤波、带短尾音；
- 不同延迟的回声使用不同音高或节奏标识；
- 压力板、门、同步成功、失败分别有独立声音；
- 环境音乐保持低密度，避免掩盖回声节奏；
- 终局信号使用前面零散声音的完整版本，形成回收。

所有第三方音乐与音效必须记录来源、作者、许可证、修改情况和署名要求。

## 14. 技术架构建议

### 14.1 核心原则

1. 模拟逻辑与显示逻辑分离；
2. 关卡数据与场景表现分离；
3. 求解器调用同一套状态转移规则；
4. 不让动画完成时机决定游戏状态；
5. 每回合状态可序列化，便于测试和撤销；
6. 先使用 GDScript；不在首版引入 C# 或原生扩展。

### 14.2 建议目录

```text
res://
├─ assets/
├─ audio/
├─ data/
│  └─ levels/
├─ scenes/
│  ├─ app/
│  ├─ gameplay/
│  ├─ levels/
│  └─ ui/
├─ scripts/
│  ├─ app/
│  ├─ simulation/
│  ├─ gameplay/
│  ├─ ui/
│  └─ tools/
├─ shaders/
└─ tests/
```

### 14.3 建议核心模块

- `SimulationState`：纯状态数据；
- `SimulationRules`：统一状态转移；
- `InputHistory`：历史输入队列；
- `EchoController`：根据延迟读取历史输入；
- `LevelDefinition`：关卡数据；
- `LevelLoader`：数据加载与校验；
- `TurnController`：驱动回合；
- `LevelSolver`：BFS/A* 求解；
- `LevelValidator`：可解性和指标；
- `SaveService`：设置与进度；
- `AudioDirector`：当前动作与回声声层；
- `SceneRouter`：主菜单、选关、游戏和结算流程。

命名和实际结构可由 Codex 在环境盘点后调整，但不能破坏模拟/显示分离原则。

## 15. AI 辅助关卡设计工作流

### 15.1 关卡提案格式

AI 生成关卡时必须输出：

- 关卡目标；
- 使用机制；
- 延迟配置；
- ASCII 网格或结构化数据；
- 预期教学点；
- 一条候选解；
- 预期最短步数区间；
- 可能的捷径；
- 失败反馈设计。

### 15.2 验证链

```text
AI 提案
→ 格式校验
→ 自动求解
→ 计算最短解与解法数量
→ 检查是否绕过目标机制
→ Codex/开发自测
→ 人类首次试玩
→ 独立体验评审
→ 正式收录
```

### 15.3 禁止事项

- 不因 AI 声称“有解”而直接收录；
- 不把未运行的示例解当成证据；
- 不用纯随机生成替代关卡设计；
- 不以最短步数高等同于好玩；
- 不允许所有关卡只改变地图尺寸而不改变推理结构；
- 不批量生成大量关卡后跳过人工试玩。
